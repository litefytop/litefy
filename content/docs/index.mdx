Litefy is a toolset built around these ideas:

## Open Code

All component source code is delivered directly. No black boxes, no irreversible abstraction layers. Want to change a button? Open `ui/button.tsx` and edit it.

## Composition-First

Every component shares a consistent API design, making behavior and interfaces predictable. You don't need to learn a new set of rules for each component.

## Lightweight Distribution

Components are installed into your project via CLI as source code. You own the code, not a dependency on an external package. No version pinning, no worries about upstream breaking changes. Your UI, your rules.

## Native-First

Interactive components like popovers and menus are built using native browser APIs (`popover`, `anchor-position`) rather than wrapping third-party libraries.

For features like `anchor-position` that have Limited availability, we provide fallback mechanisms. However, for features like `popover` that have entered Baseline, we do not provide polyfills or compatibility workarounds. This means Litefy does not support legacy browsers.

Our principle is: once a feature reaches Baseline, we ship it as-is without fallbacks. We don't want to bloat the codebase with compatibility hacks for a minority of use cases, as that would contradict our lightweight philosophy.

If your target audience still requires support for older browsers, Litefy may not be the right choice for you.

## Pure Style Components Are Not Distributed via CLI

For components that have no interaction logic and are purely style-based (e.g., separators, cards, containers), Litefy does not package them as TypeScript components. Instead, we provide copyable style code directly.

We explored multiple automation approaches: generating a fixed CSS file—but we cannot predict the location or naming of the user's CSS files; using regex injection—which can fail if the user adds comments or other content that breaks the pattern; and having the CLI auto-locate and insert—which faces the same issues.

After trying various automated solutions, we concluded that for these purely stylistic components, **manual copying is the most reliable method**. It doesn't assume anything about the user's directory structure, won't break due to custom configurations, and avoids the problem of unloading extra CSS files. It's not the most "automated" approach, but it's robust enough.

If you have a better idea, feel free to [open an issue](https://github.com/litefytop/litefy/issues).
