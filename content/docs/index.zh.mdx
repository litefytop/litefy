Litefy 是这样一套工具：

## 开放代码

所有组件源码直接交付。没有黑盒，没有不可逆的封装层。想改按钮？打开 `ui/button.tsx`，直接改。

## 组合优先

每个组件使用统一的 API 设计，行为和接口可预测。你不需要为每个新组件重新学习一套新规则。

## 轻量分发

通过 CLI 将组件源码安装到你的项目中。你拥有代码，而非依赖一个外部包。不需要锁定版本，不需要担心上游 breaking changes。你的 UI，你做主。

## 原生优先

弹窗、菜单等交互组件使用浏览器原生 API（如 `popover`、`anchor-position`）实现，而非封装第三方库。

对于 `anchor-position` 这类 Limited availability 特性，我们提供了回退机制。但对于 `popover` 这类已进入 Baseline 的特性，我们不做 polyfill 或兼容处理。这意味着 Litefy 不支持老旧浏览器。

我们的原则是：只要特性进入 Baseline，就不提供回退。我们不希望为了少数场景增加大量兼容代码，这违背了轻量的初衷。

如果你的目标用户仍需支持旧版浏览器，Litefy 可能不适合你。

## 纯样式组件不做 CLI 分发

对于没有交互逻辑、只有样式的组件（如分隔线、卡片、容器等），Litefy 选择不封装成 TS 组件，而是直接提供可复制的样式代码。

我们尝试过多种自动化方案：生成固定 CSS 文件，但无法预知用户项目中的 CSS 文件位置和命名；用正则注入，又可能因用户添加了注释或其他内容导致匹配失败；让 CLI 自动寻找并插入，也面临同样的问题。

在尝试了多种自动化方案后，我们认为：对于这类纯样式组件，**手动复制是最可靠的方式**。它不需要假设用户的目录结构，不会因为用户的个性化配置而失效，也没有引入额外 CSS 文件后的卸载问题。虽然不够“自动化”，但足够可靠。

如果你有更好的思路，欢迎[提出 Issue](https://github.com/litefytop/litefy/issues)。
